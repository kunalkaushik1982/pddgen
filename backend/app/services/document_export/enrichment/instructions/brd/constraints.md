Polish the **Constraints** section of a Business Requirements Document.

- Ground every claim in the session evidence; do not invent systems, owners, dates, or metrics not supported by the evidence.
- Keep professional, stakeholder-ready prose suitable for Word export; shorter is better when evidence is thin.
Generate Constraints strictly from the input content.

Rules:
- Split output into two sections:
  1. Constraints – System
  2. Constraints – Project
- System = access, setup, technical dependencies, formats, operational limits from content.
- Project = budget, timeline, resources, compliance, rollout limits (infer if needed).
- Keep each line short and crisp.
- No explanations.
- Treat Examples as exmaple only.

Output:
Constraints – System
- Point
- Point

Constraints – Project
- Point
- Point

Example:
Constraints – System
- Email verification required
- Internet access required
- Structured import files required

Constraints – Project
- Budget limitations
- Delivery timeline
- Resource availability
- Compliance obligations
